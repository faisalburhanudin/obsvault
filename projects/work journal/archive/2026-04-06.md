Request sequence that causes the bug

Papertrail / page-turner path
1. Client calls the demo app endpoint to start the flow.
   Example: POST /api/get-book-list
2. The demo app calls remotebrowser-dev MCP and gets back:
   - signin_id
   - url = /dpage/<signin_id>
3. On machine A, remotebrowser-dev stores flow state only in local memory:
   - active_pages
   - pending_actions
   - distillation_results
   Code: getgather/mcp/dpage.py lines 48-50
4. The browser loads GET /dpage/<signin_id>.
   Code: getgather/mcp/dpage.py lines 217-225
5. The page auto-submits POST /dpage/<signin_id>.
   Code: getgather/mcp/dpage.py lines 233-252
6. If that finishing /dpage POST hits machine A, machine A resumes the action from pending_actions[signin_id] and writes distillation_results[signin_id].
   Code: getgather/mcp/dpage.py lines 347-366
7. Meanwhile the demo app polls check_signin(signin_id), but that MCP request may hit machine B.
   Code: getgather/mcp/main.py lines 208-219
8. Machine B runs dpage_check(signin_id), which only checks its own local distillation_results dict.
   Code: getgather/mcp/dpage.py lines 160-173
9. Machine B cannot see the result written by machine A, so it keeps waiting and eventually times out.
10. Symptom: one machine finishes or closes the page, another machine keeps polling.

Datadash extra failure point
1. Datadash starts auth on one app machine.
2. The connector stores signin/session mapping only in local memory.
   Code: packages/connector/src/core/session-manager.ts line 17
3. Later /api/signin/:id or /api/poll-signin can hit another app machine.
4. That second machine may not know how to map signinId back to the session/clientId.
   Code: packages/connector/src/core/connector.ts lines 155-170

Root cause
- Multi-machine routing
- Flow state stored only in per-process memory
- No shared state and no guaranteed stickiness for signin_id/browser session
