- `mcp-getgather` intentionally reuses a stable `browser_id` in no-auth mode, derived from the app instance hostname plus `-noauth`, so the same app instance keeps the same remote browser identity across requests.
- In `flyfleet`, soft-deleting an app row only set `deleted_at` but preserved the unique `browser_id`.
- That left stale deleted rows still claiming the same `browser_id`.
- As a result, reusing that same `browser_id` could behave inconsistently: `GET` returned `404 not found` while `POST` returned `422 already exists` for the same identity.
